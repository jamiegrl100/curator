Markdown# Releases and UpdatesThis document explains how Curator Engine releases work, how “latest” downloads are managed, and how plugins/themes are versioned in Curator Marketplace.---## Curator Engine releasesCurator Engine is released as versioned packages (ZIP or tarball depending on your build).### Recommended versioningUse semantic versioning where possible:- `MAJOR.MINOR.PATCH`  Example: `0.4.0`, `0.4.1`### What belongs in an Engine releaseA clean release should include:- application code- `install/curator_schema.sql` (and optional seed)- `.env.example` (never ship a real `.env`)- docs (or links to docs)- LICENSE and READMEA release should **not** include:- `storage/` contents- logs- uploaded files- secrets---## “Latest” downloadsMost projects provide a stable “latest” download for convenience. Curator should do the same.### How “latest” works (recommended)- You upload a build artifact in Admin (Engine Builds)- You mark it as the current **Core Stable**- The download route serves that artifact:- `/download/curator-latest` → current Core Stable buildOptional:- `/download/curator-beta` → current Core Beta build- `/download/curator-nightly` → current Core Nightly build### Benefits- Users can bookmark one URL- Your marketing pages can always point to a stable link- You can roll back by switching the current build in admin---## Release notes / changelogCurator should maintain:- `CHANGELOG.md` (repo-level history)- release notes per version (in GitHub Releases or in your admin build notes)### Recommended format- Added- Changed- Fixed- Security---## Curator Marketplace releases (plugins/themes)Marketplace items (plugins/themes) are **versioned artifacts** similar to Engine.### Core rules- Each published version is **immutable**- Do not replace a ZIP for the same version number- If you need to fix something, publish a new versionExample:- `my-plugin 1.0.0` published  - bug found → publish `1.0.1`---## Update workflowsCurator should always support manual updates via ZIP upload:### Plugins- Admin → Plugins → Upload ZIP → Enable/Update### Themes- Admin → Themes → Upload ZIP → Activate/Update> Automated updates (in-app update checks) can be added later as a plugin or an enterprise feature.---## CompatibilityMarketplace items should declare compatibility:- minimum Curator version tested- highest tested version (optional)Curator Marketplace should display compatibility prominently:- “Compatible with Curator vX+”---## Rollback strategy### Engine rollback- In Engine Builds admin, select a previous build as “current stable”- `/download/curator-latest` immediately serves the previous build### Plugin rollbackManual rollback is possible by:- downloading an older published version- uploading it again in Curator admin> Marketplace can add “download older versions” later, but it’s optional for v1.---## Recommended release process (simple)1. Tag the release in GitHub (optional but recommended)2. Build the package ZIP3. Upload it to your Curator Engine Builds admin4. Set it as current Stable5. Update the marketing page version badge (or make it dynamic)6. Announce the release (marketplace/newsletter/social)---## Next- [Roadmap](./ROADMAP.md)- [Installation](./INSTALLATION.md)- [Marketplace Submission Guidelines](./MARKETPLACE_SUBMISSION_GUIDELINES.md)Copy
