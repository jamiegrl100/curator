Markdown# Contributing to Curator EngineThanks for your interest in contributing to Curator Engine.Curator is built to be a **search-first ecommerce engine** with a **theme + plugin ecosystem**. We welcome contributions that improve stability, performance, security, developer experience, and documentation.---## Ways to contribute- **Bug reports** (issues)- **Fixes** (pull requests)- **Documentation** improvements- **Themes** and **plugins**- **Security** reports (see `SECURITY.md`)---## Development principles- Keep the core **stable** and **fast**- Prefer **simple**, maintainable solutions- Use **prepared statements** and safe defaults- Avoid breaking changes unless necessary- Separate concerns:  - **Core** = business logic  - **Themes** = presentation only  - **Plugins** = extensions via hooks/routes/migrations---## Getting started (local dev)1. Clone the repository2. Configure your database and environment file (`.env` or config file used by your build)3. Install schema (`install/curator_schema.sql`) or run the installer4. Start your local server (Apache/Nginx + PHP)See:- `docs/INSTALLATION.md`- `docs/QUICK_START.md`---## Reporting bugsWhen opening an issue, include:- What you expected to happen- What actually happened- Steps to reproduce- Logs/error messages (sanitize secrets)- Environment:  - OS  - PHP version  - MySQL/MariaDB version  - Web server (Apache/Nginx)  - Curator version---## Pull requests### Before you submit- Keep PRs focused (one change per PR when possible)- Run basic smoke tests locally- Update docs if behavior changes- Avoid formatting-only PRs unless requested### PR checklist- [ ] No secrets committed (`.env`, API keys, passwords)- [ ] Uses prepared statements for DB access- [ ] No new external network calls without review- [ ] No breaking changes without discussion- [ ] Documentation updated (if needed)---## Coding standards- PHP: follow common PSR-style conventions- Keep code readable (small functions, clear names)- Prefer explicit error handling and clear messages- Comments should explain **why**, not **what**---## Plugins and themesIf you’re contributing a plugin or theme to the ecosystem:- Follow the manifest schemas:  - `docs/PLUGINS.md`  - `docs/THEMES.md`- Do not include:  - obfuscated code  - remote code execution  - self-updaters that bypass marketplace review- Provide:  - clear README  - usage instructions  - compatibility infoIf you plan to submit to **Curator Marketplace**, your plugin/theme will go through automated checks and manual review.---## Versioning- Use semantic versioning where possible: `MAJOR.MINOR.PATCH`- Marketplace releases should include a changelog entry.---## CommunityIf you don’t know where to start:- open an issue describing what you want to build- or submit a small doc improvement PR first---## LicenseBy contributing, you agree that your contributions will be licensed under the project’s license (see `LICENSE`).Copy
