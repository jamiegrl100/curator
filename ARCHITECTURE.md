Markdown# ArchitectureThis document explains Curator Engine at a high level: how requests flow through the system, where themes/plugins fit, and how the marketplace pipeline works (if enabled).> This is intentionally practical. It’s meant to help contributors understand where code belongs.---## High-level overviewCurator is composed of these major layers:1. **HTTP routing** (maps requests to handlers/controllers)2. **Core application** (business logic: catalog, search, orders, downloads)3. **Theme system** (presentation/templates)4. **Plugin system** (extensions via hooks/routes/migrations/assets)5. **Storage layer** (uploads, releases, logs, temp files)6. **Marketplace pipeline** (submission → checks → review → publish) *(marketplace builds only)*---## Request lifecycleA typical request follows this path:1. Web server receives request (Apache/Nginx)2. Front controller (often `public/index.php`) boots the app3. Core loads configuration (settings + env)4. Core loads enabled plugins (unless safe mode)5. Plugins register hooks/routes/admin pages6. Router matches the request7. Controller builds a `$data` payload8. Theme renderer selects templates and renders output9. Response is returned---## RoutingCurator uses a router that supports:- method routing (GET/POST)- route parameters (e.g., `/marketplace/{type}/{slug}`)- middleware checks (auth/admin)### Route namespaces- Public routes:  - `/`  - `/marketplace`  - `/docs`  - `/support`- Account routes:  - `/account/*`- Admin routes:  - `/admin/*`- Plugin routes:  - `/p/<plugin-slug>/*` (required prefix)Plugin routes are prefixed to reduce collisions and keep core stable.---## Core modulesCurator core typically includes:### Catalog + content- products- categories/tags- product images/files- product page rendering### Search- keyword search (scalable patterns like FULLTEXT or indexed table)- filters (category/tags/price)- sorting (relevance/newest/popular/price)### Commerce (digital-first core)- carts + cart items- orders + order items- hosted checkout flows- order lifecycle (pending → paid → refunded)### Fulfillment (downloads)- download tokens per purchased file- expiry/max downloads/revocation- “download all as ZIP” (optional feature)### Email- SMTP settings (optional)- queued sending + retries- transactional templates### Tax (optional module)- tax-by-location (manual zones/rates)- shipping disabled by default in digital-first builds---## Theme systemThemes are presentation-only.### Responsibilities- layout templates- page templates- partials (header/footer/nav)- CSS/JS assets### Non-responsibilities- direct database queries- payment logic- permission checks### Template resolutionCore selects a template key (e.g., `home`, `search`, `product`) and renders using the active theme. If a template is missing, the renderer may fall back to the default theme.See: [Theme Development](./THEMES.md)---## Plugin systemPlugins extend Curator through:- hooks (actions/filters)- routes (prefixed)- admin pages- migrations (on enable)- assets (CSS/JS)### Safe modeCurator should support safe mode so a broken plugin cannot brick the site:- safe mode disables plugin loading- admins can disable the failing pluginSee: [Plugin Development](./PLUGINS.md) and [Hooks Reference](./HOOKS_REFERENCE.md)---## Storage layoutCurator stores files outside web root. A common structure:- `storage/uploads/` (images, editor assets)- `storage/files/` (digital product files)- `storage/tmp/` (temporary zips)- `storage/logs/` (application logs)- `storage/marketplace/` (marketplace submissions/published artifacts)Nothing under `storage/` should be directly web-accessible. Serve downloads through controlled endpoints.---## Marketplace pipeline (marketplace builds)Curator Marketplace adds:- developer submissions (listings + releases)- automated checks (zip safety, manifest validation, basic security heuristics)- reviewer queue (approve/reject/changes requested)- publishing (moves artifacts into published storage)- public listing pages and downloads### Release immutabilityPublished releases should be treated as immutable:- each version is a unique artifact- do not replace ZIP contents for the same versionSee: [Marketplace Submission Guidelines](./MARKETPLACE_SUBMISSION_GUIDELINES.md)---## Security model (quick summary)- Prepared statements for DB access- CSRF protection on all state-changing actions- Strict path validation for ZIP extraction (zip-slip prevention)- Download authorization checks (order ownership + token rules)- Admin-only access to dangerous actions (publishing, settings, exports)See: `SECURITY.md`---## Where to put new code (rules of thumb)- **Core business logic** → core module/services- **Presentation/layout** → theme templates- **Optional features** → plugins- **Distribution/review** → marketplace pipeline- **Admin tools** → admin controllers/pages + servicesIf you’re unsure, open an issue and describe the feature and intended behavior.---## Next- [Hooks Reference](./HOOKS_REFERENCE.md)- [Plugin Development](./PLUGINS.md)- [Theme Development](./THEMES.md)- [Roadmap](./ROADMAP.md)Copy
